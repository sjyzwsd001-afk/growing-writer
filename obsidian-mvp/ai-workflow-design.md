# AI 工作流设计

这是一版面向 Obsidian 写作助手 MVP 的 AI 工作流。

目标是让系统围绕一条稳定链路工作：

1. 读历史材料
2. 召回相似案例
3. 调用已确认规则
4. 先诊断再生成
5. 接收反馈并沉淀新规则

## 1. 工作流总览

建议把系统拆成 5 个阶段：

1. `材料理解`
2. `任务理解`
3. `写前诊断`
4. `提纲与初稿生成`
5. `反馈学习`

这 5 个阶段不是一个大 prompt 一次完成，而是分步执行。

## 2. 阶段一：材料理解

输入：

- `materials/` 中的历史材料
- `rules/` 中已有规则
- `profiles/` 中已有写作画像

处理目标：

- 识别文档类型
- 提取结构特征
- 提取风格特征
- 提取候选规则

建议输出：

- 文档摘要
- 文档结构标签
- 风格观察
- 候选规则列表

### 建议做法

每篇材料导入时，不直接扔给生成模型当上下文，而是先做一层结构化整理：

- 标题
- 文体类型
- 面向对象
- 核心结构
- 高质量段落
- 可复用表达
- 候选规则

这样后续召回时不会把整篇长文都塞进去。

## 3. 阶段二：任务理解

输入：

- `tasks/` 中当前任务
- 用户补充的新素材

系统先判断：

- 这是什么文体
- 面向谁
- 目标是什么
- 信息是否充分
- 更像哪一类历史任务

建议输出一个标准化任务对象：

```yaml
task_type:
audience:
scenario:
goal:
must_include:
constraints:
raw_facts:
missing_info:
```

这一步的作用是把“散乱素材”变成“可写任务”。

## 4. 阶段三：召回与写前诊断

这是第一版最关键的一层。

系统不要拿到素材就直接写，而是先做两类召回：

1. `相似材料召回`
2. `规则召回`

### 4.1 相似材料召回

优先召回：

- 同文体材料
- 同受众材料
- 同场景材料
- 被标记为高质量的材料

建议召回 3-5 篇，不要太多。

召回结果需要压缩成：

- 典型结构
- 可借鉴表达
- 常见展开顺序
- 结尾方式

### 4.2 规则召回

规则召回分两层：

- `强匹配规则`
  比如明确适用于某文体、某受众
- `弱匹配规则`
  比如通用表达要求、通用结构要求

规则优先级建议：

1. 已确认规则
2. 高置信度候选规则
3. 通用规则

### 4.3 写前诊断

这一步是你的产品区别于普通写作工具的关键。

系统先输出：

- 当前信息是否足够成稿
- 缺失的关键因素
- 建议采用的结构
- 哪些规则将被使用
- 哪些历史材料最值得参考

建议用户先看诊断，再决定是否生成。

诊断结果可以写成：

```markdown
# 写前诊断

## 当前判断
- 任务类型：
- 面向对象：
- 建议结构：

## 信息缺口
- 

## 已命中规则
- 

## 参考材料
- 
```

## 5. 阶段四：提纲与初稿生成

生成不要一次性出最终稿，建议拆成 3 步。

### 5.1 先生成提纲

输入：

- 标准化任务对象
- 相似材料摘要
- 已命中规则
- 写作画像摘要

输出：

- 一级结构
- 每部分承担的功能
- 每部分要写哪些事实

提纲阶段的目标不是语言漂亮，而是结构正确。

### 5.2 再生成初稿

输入：

- 已确认提纲
- 关键事实
- 已命中规则
- 参考表达

输出：

- 第一版正文

这一阶段建议要求模型：

- 不编造事实
- 缺失信息明确标注
- 优先使用已确认规则
- 尽量贴近已沉淀风格

### 5.3 最后做统一修订

不是重写，而是做一轮轻修订：

- 语言统一
- 段落衔接
- 逻辑顺序检查
- 是否漏写关键因素
- 是否违反禁忌规则

这一轮更像“审稿器”。

## 6. 阶段五：反馈学习

这一层决定系统会不会越来越好。

用户修改后，系统不要直接“记住整段新文字”，而要先判断反馈类型。

建议分成 6 类：

1. `措辞问题`
2. `结构问题`
3. `顺序问题`
4. `逻辑问题`
5. `信息缺失`
6. `场景不匹配`

然后再判断：

- 这次反馈是一次性修改
- 还是长期偏好
- 是否值得升级为规则

### 6.1 反馈学习输出

每次学习后建议产出：

- 修改摘要
- 反馈分类
- 候选规则
- 建议适用场景
- 是否需要人工确认

### 6.2 为什么不能直接自动入库

因为很多修改是场景性的，比如：

- 这次领导特别关注风险
- 这次篇幅必须压缩
- 这次项目需要强调政策背景

这些都不一定能变成长期规则。

所以建议机制是：

- 默认先进 `candidate`
- 人工确认后再进入 `confirmed`

## 7. MVP 的提示词链路

第一版不需要复杂 agent 系统，可以先做 4 个固定步骤。

### Prompt 1：任务解析

作用：

- 识别任务类型、受众、目标、约束
- 提取缺失信息

输入：

- 当前任务素材

输出：

- 标准化任务对象

### Prompt 2：材料与规则整理

作用：

- 总结召回材料的结构和风格
- 挑出适用规则

输入：

- 召回材料摘要
- 规则卡片摘要

输出：

- 建议结构
- 命中规则
- 参考表达

### Prompt 3：提纲生成

作用：

- 输出结构稳定的提纲

输入：

- 任务对象
- 命中规则
- 参考材料摘要

输出：

- 提纲

### Prompt 4：正文生成与审校

作用：

- 基于提纲生成初稿
- 再做轻量审校

输入：

- 提纲
- 素材
- 规则
- 风格摘要

输出：

- 初稿
- 风险提醒

## 8. 一版很实用的上下文拼装原则

为了避免 token 爆炸，建议按这个顺序拼上下文：

1. 当前任务
2. 当前命中的已确认规则
3. 当前最相似的 3 篇材料摘要
4. 写作画像摘要
5. 必要的候选规则

不要直接塞：

- 全部历史材料
- 全部规则
- 全部反馈记录

系统需要的是“精召回”，不是“全量投喂”。

## 9. 第一版最小技术实现建议

如果我们先做一个可跑版本，不需要太复杂。

最小实现可以是：

- Obsidian 存储所有 Markdown
- 检索插件负责相似材料召回
- 一个本地脚本或轻量服务负责：
  - 读取任务文件
  - 读取命中规则
  - 读取召回材料摘要
  - 拼装 prompt
  - 输出结果回写到任务文件

反馈学习也先不用做全自动，可先半自动：

- 读取反馈模板
- 生成候选规则建议
- 由人工确认后写入规则文件

## 10. 我建议你第一版先验证的 3 个问题

第一版先不要追求全自动，先验证这 3 个核心问题：

1. 历史材料召回后，是否明显提升初稿质量
2. 规则卡片是否比单纯 RAG 更稳定
3. 反馈沉淀后，后续任务是否真的更贴近你的写法

只要这三个问题成立，这个产品就值得继续做。
